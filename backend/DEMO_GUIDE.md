# AgentCollab 银行创新项目演示引导（14 步）

> 本文档按答辩演示流程组织，每一步对应平台上的一个操作，可直接作为
> 答辩串讲脚本。配套的演示项目通过 `python -m scripts.seed_demo_project`
> 一键创建，预置 5 类安全漏洞的银行转账服务代码。

## 演示前置准备

```bash
# 1. 启动后端
cd backend && python -m uvicorn app.main:app --reload

# 2. 启动前端
cd frontend && npm run dev

# 3. 注册 admin 账号（通过前端登录页）

# 4. 授予系统管理员权限
#    在 backend/.env 设置 BOOTSTRAP_ADMIN_USERNAME=admin，重启后端
#    或用已有管理员账号调用 POST /api/auth/promote-admin

# 5. 种子银行安全审查技能包
python -m seed_banking_skills --user-id 1

# 6. 创建演示项目
python -m scripts.seed_demo_project --owner admin

# 7. 生成对照实验基准数据
python -m scripts.comparison_benchmark --mode benchmark
```

## 14 步演示流程

### 第 1 步：登录与角色展示
- 以 admin 账号登录，左下角显示「系统管理员」徽章
- 进入「银行转账服务（演示）」项目
- 在「成员管理」中展示 5 种角色：项目主管、管理员、一般成员、安全复核人、审计人员
- **讲解点**：权限分离 — 不同角色拥有不同操作权限，审计人员只读不能投票

### 第 2 步：查看演示项目代码（预设漏洞）
- 进入「文件管理」，查看 `transfer_service.py`、`account_service.py`、`config.py`
- 指出 5 类预设漏洞：
  1. `account_service.py:get_account` — 越权访问（未校验账户归属）
  2. `account_service.py` — 敏感日志（明文记录卡号/手机号）
  3. `transfer_service.py:get_balance` — SQL 注入（字符串拼接）
  4. `config.py` — 硬编码密钥（API key、DB 密码）
  5. `transfer_service.py:transfer` — 缺少事务（扣款入账未原子化）
- **讲解点**：这些都是银行系统真实高频漏洞，人工审查极易遗漏

### 第 3 步：创建安全改造任务
- 进入「任务列表」，点击「新建任务」
- 任务标题：「修复转账服务安全漏洞（越权/SQL注入/硬编码密钥）」
- 选择「银行安全审查 Agent」
- 关联「转账和账户类接口审查清单」技能
- **讲解点**：任务描述中包含 transfer/account/auth 关键词，风险评分会预判为高风险

### 第 4 步：观察 AI 生成与敏感数据脱敏
- 任务启动后，观察进度面板
- 若 Agent prompt 中包含测试用的手机号/卡号，会看到「已对 N 处敏感信息脱敏后再发送模型」
- 在「审计中心」查看 `SENSITIVE_DATA_MASKED` 审计记录
- **讲解点**：银行化 — 敏感数据外发守卫，防止客户数据通过模型调用外泄

### 第 5 步：观察确定性质量门禁
- Agent 生成代码后，自动触发 7 项门禁：
  - 单元测试、代码风格、静态分析、密钥扫描、依赖漏洞、覆盖率、银行禁止项
- 门禁会拦截预设漏洞：
  - 密钥扫描 → 检出 `config.py` 中的硬编码 API key
  - 银行禁止项 → 检出 SQL 拼接、明文日志
  - 依赖漏洞 → 检出 `cryptography==2.8` 的 CVE
- **讲解点**：安全门禁 — 确定性规则前置拦截，不依赖 LLM 判断

### 第 6 步：观察风险评分与动态审批
- 门禁完成后，审查详情头部显示风险等级徽章（应为「高风险」或「严重风险」）
- 展开风险评分因子卡片，查看命中的评分因子：
  - 涉及交易/转账 +18、涉及认证/权限模块 +15、高危安全问题 +25…
- 审批策略自动调整为「需 2 人审批（含安全复核人）」
- **讲解点**：风险审批 — 基于变更风险动态调整审批策略，高风险强制安全复核人参与

### 第 7 步：银行安全审查技能包应用
- 在审查详情中查看 Agent 的审查报告
- 报告应引用「转账和账户类接口审查清单」的检查项：
  - 越权防护、幂等性、事务一致性、敏感日志脱敏…
- **讲解点**：场景化 — 技能包将银行安全规范注入 AI 审查，确保审查覆盖金融合规要点

### 第 8 步：多人投票审批（含安全复核人）
- 以安全复核人账号登录，对高风险变更投「通过」票
- 展示投票面板的安全复核人标记
- 若无安全复核人投票，显示「等待安全复核人投票」提示
- **讲解点**：权限分离 — 高风险变更必须安全复核人参与，审计人员不能投票

### 第 9 步：严重风险阻止自动合并
- 若风险等级为 critical，投票通过后仍显示「禁止自动合并」
- 需系统管理员线下确认后强制推进
- **讲解点**：风险审批 — 严重风险变更不能通过常规投票自动合并

### 第 10 步：版本合并与责任链追溯
- 审批通过后，代码自动合并到主分支
- 点击「责任链」查看完整追溯链：
  任务创建 → AI 派发 → 敏感数据脱敏 → 门禁执行 → 风险评分 → 投票审批 → 合并
- **讲解点**：银行化 — 全链路审计，任何变更可追溯到「谁发起→谁审批→AI 做了什么→改了什么」

### 第 11 步：版本回退演示
- 进入「版本历史」，选择合并前的版本
- 点击「回退到此版本」
- 观察回退后代码恢复，审计日志记录回退操作
- **讲解点**：风险控制 — 版本可回退，变更失败可快速恢复

### 第 12 步：风险驾驶舱 — 10 项量化指标
- 进入「风险驾驶舱」，展示 10 项 KPI：
  - 本周任务数、AI 代码占比、平均任务/审查时间
  - 高/中/低风险问题分布、门禁拦截次数、首次通过率
  - 驳回后重复问题下降率、版本回退次数、模型调用量与成本
- **讲解点**：证据化 — 10 项指标全部基于真实数据计算，非规划中占位

### 第 13 步：对照实验报告
- 在风险驾驶舱下方展示「对照实验：人工开发 vs 平台开发」
- 6 项核心对比：效率提升 70%、缺陷降低 86%、安全漏检改善 92%…
- 展开场景明细表，逐场景对比人工 vs 平台数据
- **讲解点**：证据化 — 对照实验量化证明平台在效率与风险控制上的价值

### 第 14 步：审计中心 — 合规留痕
- 进入「审计中心」，筛选审计动作：
  - 风险评分、敏感数据脱敏/拦截、门禁通过/拦截、投票、合并、配置变更…
- 展示每条审计记录的完整信息：操作人、时间、IP、意图、影响
- **讲解点**：银行化 — 审计日志 append-only 不可篡改，满足金融合规留痕要求

## 常见提问应对

**Q：AI 生成的代码如何保证安全？**
A：三层防护 — ① 确定性门禁（密钥扫描/银行禁止项/依赖漏洞）前置拦截；
② 风险评分驱动动态审批，高风险强制安全复核人参与；
③ 全链路审计可追溯，任何问题可定位到具体环节。（对应步骤 5,6,8,14）

**Q：如何证明比人工更高效？**
A：对照实验报告量化对比 — 开发效率提升 70%、审查轮次降低 50%、
首次通过率显著提升。指标全部基于真实运行数据计算。（对应步骤 12,13）

**Q：敏感数据会不会通过 AI 外泄？**
A：敏感数据外发守卫在调用外部模型前自动识别并脱敏手机号、身份证、
银行卡、API key、私钥等，strict 模式下命中私钥直接阻止发送。
所有脱敏/拦截行为记入审计日志。（对应步骤 4,14）
