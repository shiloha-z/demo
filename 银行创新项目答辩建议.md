# AgentCollab 银行创新项目答辩与优化建议

> 文档用途：帮助团队梳理项目痛点、创新点、改进方向、演示方案和答辩话术。  
> 项目建议定位：**面向银行研发场景的可控、可审、可回退的 AI 协作治理平台。**

---

## 一、项目概述

AgentCollab 是一个多 Agent 协作代码生成与审查平台。

用户提交研发任务后，平台能够完成以下流程：

1. AI Agent 根据需求生成或修改代码；
2. 代码审查 Agent 检查逻辑、规范和潜在缺陷；
3. 安全审查 Agent 检查注入、越权、敏感信息泄露等风险；
4. 汇总 Agent 生成结构化审查报告；
5. 人工复核并进行通过、驳回或终止操作；
6. 审批通过后合并代码并记录版本；
7. 出现问题时可以追溯操作过程并回退版本。

项目的核心并不是“让 AI 自动写代码”，而是：

> 让 AI 在银行现有的权限、复核、审计和变更管理框架下，安全地参与软件研发。

## 二、银行研发场景中的主要痛点

### 1. 通用 AI 编程工具缺少过程治理

通用 AI 编程工具通常关注代码生成效率，但在银行场景中还需要回答：

- 谁向 AI 提出了什么要求？
- AI 查看和修改了哪些文件？
- AI 为什么做出这项修改？
- 修改经过了哪些安全检查？
- 谁审查并批准了这项变更？
- 出现问题后能否快速定位和恢复？

如果上述问题无法回答，AI 生成的代码就很难直接进入银行研发流程。

### 2. 人工代码审查效率较低

传统代码审查依赖研发人员逐行检查，容易出现：

- 重复性检查耗费大量时间；
- 审查质量受个人经验影响；
- 安全问题可能被遗漏；
- 多人审查意见难以统一汇总；
- 项目规模扩大后审查人员压力持续增加。

### 3. 安全规范依赖个人记忆

银行内部通常存在大量开发规范、安全要求和历史经验，但这些知识可能散落在：

- 制度文件；
- 项目文档；
- 历史缺陷记录；
- 审查人员个人经验；
- 聊天记录和会议纪要。

新员工或实习生很难在短时间内掌握全部要求，同类问题也可能在不同项目中重复出现。

### 4. AI 修改代码存在不可控风险

如果 AI 可以直接修改主分支，可能造成：

- 多个任务相互覆盖；
- 未审查代码进入正式版本；
- 错误修改影响其他开发人员；
- 安全问题随代码一同发布；
- 出现故障时难以恢复。

### 5. 变更责任链不完整

传统开发工具通常分别记录需求、代码、审批和版本信息，数据分散在不同系统中，很难快速还原：

> 谁提出需求 → AI 执行了什么 → 发现了哪些风险 → 谁作出审批 → 最终产生了什么影响。

## 三、项目核心创新点

### 1. 多 Agent 职责分离

平台将不同职责交给不同 Agent：

| Agent 角色 | 主要职责 |
|---|---|
| 代码生成 Agent | 分析任务并生成或修改代码 |
| 代码审查 Agent | 检查逻辑、可读性、规范和潜在 Bug |
| 安全审查 Agent | 检查注入、越权、敏感信息和不安全配置 |
| 汇总 Agent | 汇总审查结论并按照严重程度输出报告 |

这种方式不是让一个模型同时“写代码并给自己打分”，而是通过职责分离形成交叉检查。

### 2. AI 与人工协同决策

AI 负责生成、检查和提供建议，最终决策仍由人工完成。

平台支持：

- 指定多名复核人员；
- 设置法定通过人数；
- 配置驳回一票否决；
- 填写驳回原因并要求 Agent 重新修改；
- 未经批准的代码不能进入主分支。

这与银行业务中的“经办—复核”思想具有较高契合度。

### 3. 隔离式代码执行

每项任务都在独立 Git 分支和工作树中执行：

```text
主分支
├── 任务 A 独立工作树
├── 任务 B 独立工作树
└── 任务 C 独立工作树
```

带来的价值包括：

- 不同任务互不影响；
- AI 无法直接污染主分支；
- 人工可以在合并前查看完整 Diff；
- 驳回后可以继续在原任务分支修改；
- 任务终止后可以清理对应分支。

### 4. 全链路审计责任链

平台将以下信息串联起来：

- 人工创建任务；
- 向 AI 下达的指令；
- Agent 执行过程；
- 审查意见和投票；
- 驳回反馈；
- 代码合并；
- 冲突处理；
- 版本回退。

通过责任链可以还原一次变更的完整过程，提高问题定位和责任追溯效率。

### 5. 四层经验记忆

平台采用任务、Agent、项目和全局四层记忆：

| 记忆层级 | 记录内容 |
|---|---|
| 任务记忆 | 当前任务的上下文、进度、错误和决策 |
| Agent 记忆 | 某个 Agent 的历史经验和常见错误 |
| 项目记忆 | 项目架构、历史审查结论和项目规范 |
| 全局记忆 | 可以跨项目复用的通用安全与开发经验 |

例如，某次代码因为“敏感信息写入日志”被驳回后，平台可以保存该经验。后续出现相似任务时，Agent 能够主动规避同类问题。

### 6. 可回退的变更闭环

代码只有在审批通过后才进入合并队列，并生成版本记录。

如果新版本出现问题，可以：

1. 定位对应任务；
2. 查看当时的指令、审查和审批记录；
3. 找到受影响版本；
4. 执行可审计的版本回退。

这使平台形成“生成—审查—审批—合并—追溯—恢复”的完整闭环。

## 四、与普通 AI 编程工具的区别

| 对比维度 | 普通 AI 编程工具 | AgentCollab |
|---|---|---|
| 主要目标 | 提高个人编码速度 | 提升团队研发效率并控制 AI 风险 |
| 执行方式 | 单一助手生成代码 | 多 Agent 分工协作 |
| 代码隔离 | 取决于使用者操作 | 每个任务独立分支和工作树 |
| 安全审查 | 通常需要人工额外完成 | 独立安全 Agent 自动审查 |
| 人工审批 | 通常没有完整流程 | 多人投票、驳回反馈、人工决策 |
| 审计能力 | 对话和代码记录相对分散 | 指令、执行、审查、审批和影响形成责任链 |
| 经验复用 | 主要依赖个人上下文 | 四层记忆和技能仓库 |
| 故障恢复 | 依赖外部版本工具 | 平台内置版本记录与回退 |

答辩时不建议只说“使用了多个 Agent”，而应强调：

> 我们创新的重点不是模型数量，而是把 AI 能力嵌入银行可控的研发治理流程。

## 五、建议新增或加强的功能

### P0：答辩前必须完成

#### 1. 修复前端构建问题

当前后端回归测试可以通过，但前端构建仍存在 TypeScript 错误。正式演示前应保证：

- 前端能够正常构建；
- 后端服务能够正常启动；
- 核心演示流程可以连续运行；
- 准备一套备用演示数据；
- 外部模型不可用时有录屏或模拟数据兜底。

#### 2. 关闭无鉴权调试接口

目前调试管理接口不具备完整访问控制。银行场景下建议：

- 生产模式不注册调试路由；
- 管理接口增加系统管理员鉴权；
- 系统设置接口仅允许管理员访问；
- 配置变更记录真实操作人；
- 不在前端、日志或接口响应中暴露密钥。

#### 3. 完善角色权限

建议至少区分：

| 角色 | 建议权限 |
|---|---|
| 项目负责人 | 管理成员、配置审批策略、最终管理项目 |
| 项目管理员 | 管理任务、评审人和项目配置 |
| 开发成员 | 创建任务、查看代码和提交修改建议 |
| 安全复核人 | 查看安全报告并参与高风险审批 |
| 审计人员 | 只读查看审计、版本和责任链 |

普通成员不应绕过审查流程直接修改主分支。

### P1：最值得增加的银行特色

#### 1. 银行研发安全策略门禁（已实现）

在代码合并前执行确定性检查：

- 单元测试；
- 代码格式与规范检查；
- SQL 注入、命令注入等静态扫描；
- 硬编码密钥扫描；
- 依赖漏洞检查；
- 测试覆盖率检查；
- 银行内部禁止项检查。

只有“AI 审查 + 确定性工具检查 + 人工审批”全部通过后才允许合并。

当前平台已将七项检查放在人工审批之前，检查结果会持久化并实时展示。任一必需检查失败或未配置都会禁止投通过票。系统进一步区分“代码问题”和“平台环境问题”：测试失败、安全发现等代码问题可一键打回 Agent；命令未配置、检查工具缺失等平台问题由管理员处理，修复后直接对原提交重新检查，避免 Agent 无效重跑。

#### 2. 经办—复核分离

建议增加以下规则：

- 任务发起人不能独自批准自己的任务；
- Agent 创建者不能作为唯一复核人；
- 高风险任务至少需要两人复核；
- 涉及认证、交易、账户、加密的修改必须经过安全人员；
- 驳回意见不能为空；
- 审批策略一旦开始投票便不能随意修改。

#### 3. 基于风险的动态审批

不再只使用固定的“通过百分比”，而是根据变更风险自动调整审批要求。

风险评分可以考虑：

- 修改文件数量；
- 代码变更行数；
- 是否涉及认证或权限模块；
- 是否涉及交易、账户或客户信息；
- 是否修改数据库结构；
- 是否新增外部依赖；
- 是否发现高危安全问题；
- 是否缺少自动化测试。

示例：

| 风险等级 | 审批策略 |
|---|---|
| 低风险 | 1 人审批 + 自动检查通过 |
| 中风险 | 2 人审批 |
| 高风险 | 2 人审批 + 安全复核人参与 |
| 严重风险 | 禁止自动合并，必须线下确认 |

#### 4. 防止敏感数据外发

银行评委很可能询问：“代码和数据是否会发送给外部大模型？”

建议补充或明确：

- 支持银行内网模型或私有化模型；
- 外部模型地址采用白名单；
- 发送模型前进行密钥、账号、手机号等敏感信息识别；
- 敏感内容自动脱敏或阻止发送；
- 记录模型名称、版本和调用时间；
- 禁止使用真实客户数据进行演示。

### P2：时间充足时增加

#### 1. 银行规范技能包

在技能仓库中预置：

- Java/Python 安全编码规范；
- API 权限校验规范；
- 数据库访问规范；
- 日志脱敏规范；
- 密码与密钥管理规范；
- 转账和账户类接口审查清单。

这比单纯展示公共技能导入更贴合银行业务。

#### 2. 风险驾驶舱

增加可视化指标：

- 本周任务数量；
- AI 生成代码占比；
- 平均任务处理时间；
- 平均人工审查时间；
- 高、中、低风险问题数量；
- 自动门禁拦截次数；
- 首次通过率；
- 驳回后重复问题下降率；
- 版本回退次数；
- 模型调用量和估算成本。

#### 3. 审计防篡改

如果希望强调审计可信度，可以进一步增加：

- 每条审计记录包含上一条记录的哈希；
- 对关键字段生成摘要；
- 定期生成审计批次签名；
- 审计记录只允许追加，不允许业务接口删除；
- 支持导出审计报告。

在完成前，不建议直接宣称平台审计记录“绝对不可篡改”。

## 六、建议使用的量化指标

创新项目不能只展示功能，还需要说明产生了什么价值。

建议选择一组典型任务进行人工与平台对照测试。

| 指标 | 计算方式 |
|---|---|
| 平均交付时间 | 从任务创建到审批合并的平均时间 |
| 人工审查耗时 | 审查人员实际投入时间 |
| 首次通过率 | 首次审查直接通过的任务数 ÷ 总任务数 |
| 缺陷拦截率 | 上线前发现问题数 ÷ 全部发现问题数 |
| 重复问题下降率 | 使用记忆前后同类问题出现次数对比 |
| AI 代码采用率 | 最终保留的 AI 代码行数 ÷ AI 生成代码行数 |
| 自动门禁拦截量 | 测试、安全和规范检查拦截次数 |
| 回退恢复时间 | 从发现问题到恢复稳定版本的耗时 |
| 单任务成本 | 模型调用、人工审查和运行资源的估算成本 |

不要只强调“审查通过率高”。通过率过高也可能让评委怀疑审查标准不够严格。

更推荐展示：

> 平均处理时间下降、上线前风险拦截增加、人工重复劳动减少、同类错误复发率降低。

## 七、推荐现场演示案例

### 案例：转账查询接口安全改造

#### 演示目标

通过一个任务完整展示代码生成、安全审查、人工复核、驳回学习、自动门禁、责任链和版本回退。

#### 演示流程

1. 创建项目“银行转账服务”；
2. 创建任务：“新增转账记录查询接口”；
3. Agent 生成接口代码；
4. 代码中预设一个越权访问或敏感日志问题；
5. 安全 Agent 识别问题并标记风险等级；
6. 人工查看代码 Diff 和安全报告；
7. 复核人填写意见并驳回；
8. Agent 根据意见重新修改；
9. 自动测试和安全门禁通过；
10. 两名复核人投票批准；
11. 任务进入合并队列并生成版本；
12. 打开责任链，展示完整操作记录；
13. 创建一个相似任务，展示 Agent 已主动规避上次问题；
14. 可选：模拟新版本异常并展示版本回退。

#### 推荐预设的风险点

演示时选择一到两个即可：

- 未校验当前用户是否有权查看目标账户；
- 将银行卡号或身份证号完整写入日志；
- SQL 查询使用字符串拼接；
- 在代码中硬编码数据库密码；
- 接口缺少金额范围或分页限制；
- 异常信息直接返回内部堆栈。

不要在一次演示中放入太多漏洞，以免主线不清晰。

## 八、答辩汇报结构

建议按照以下顺序制作 PPT。

### 第1页：项目名称

**AgentCollab——面向银行研发的可控 AI 协作治理平台**

副标题可以使用：

> 让 AI 提升研发效率，同时确保每一次修改可控、可审、可追溯、可回退。

### 第2页：背景与痛点

- 通用 AI 编程工具难以满足银行治理要求；
- 人工代码审查成本较高；
- 安全规范和历史经验难以复用；
- AI 修改代码存在越权、泄密和不可追溯风险；
- 需求、代码、审查、审批和版本信息相互割裂。

### 第3页：解决方案

展示核心闭环：

```text
创建任务
   ↓
代码生成 Agent
   ↓
代码审查 + 安全审查
   ↓
自动化安全门禁
   ↓
多人复核
   ↓
合并与版本记录
   ↓
审计追溯与经验沉淀
```

### 第4页：核心创新

- 多 Agent 职责分离；
- AI 与人工协同决策；
- 任务级隔离工作树；
- 基于风险的动态审批；
- 全链路责任追踪；
- 四层经验记忆。

### 第5页：银行场景适配

- 经办复核分离；
- 角色权限控制；
- 安全策略门禁；
- 敏感数据保护；
- 私有化模型适配；
- 审计和版本回退。

### 第6页：现场演示

演示“转账查询接口安全改造”完整闭环。

### 第7页：量化成效

展示人工方式与平台方式的对比数据。

### 第8页：后续规划

- 对接银行统一身份认证；
- 接入内部代码仓库和流水线；
- 对接内部安全扫描工具；
- 使用银行内网大模型；
- 完善模型风险治理；
- 开展更多项目试点。

## 九、推荐答辩话术

### 30 秒项目介绍

> 我们的项目不是简单让 AI 自动写代码，而是解决 AI 进入银行研发流程以后如何被安全使用的问题。平台通过多 Agent 分工、独立代码工作树、自动安全审查、多人复核、全链路审计和版本回退，让 AI 的每一次修改都有明确的发起人、执行过程、审查结论和最终责任人，从而在提升研发效率的同时控制风险。

### “你们和 Copilot 有什么区别？”

> Copilot 更偏向个人编码辅助，我们的平台关注团队级研发治理。我们不仅生成代码，还管理 AI 的执行权限、代码隔离、安全检查、多人审批、责任追溯、经验沉淀和版本恢复。创新点在于把 AI 能力放进银行可以接受的受控流程中。

### “为什么需要多个 Agent？”

> 如果同一个 Agent 同时写代码并审查自己，容易形成单一判断偏差。我们将生成、质量审查、安全审查和结果汇总分给不同角色，使其从不同角度交叉检查，再由人工作出最终决定。

### “AI 出错怎么办？”

> AI 的修改首先发生在隔离工作树，不会直接影响主分支；之后还要经过安全审查、自动化门禁和人工复核。即使合并后发现问题，也可以通过责任链定位对应任务，并利用版本记录进行回退。

### “客户数据会不会发送到外部？”

> 当前项目属于创新原型，正式落地时将优先接入银行内网或私有化模型，并增加敏感信息识别、脱敏、模型地址白名单和调用审计。演示和试点阶段不使用真实客户数据。

### “项目是否已经可以生产使用？”

> 当前已经完成关键流程和技术可行性验证，但银行生产部署还需要对接统一身份认证、内部代码仓库、安全扫描平台和私有化模型，并进一步完善权限、审计防篡改和高可用能力。我们将其定位为具备完整核心闭环的创新原型。

## 十、答辩时应避免的表述

在没有充分验证前，不建议直接使用以下说法：

- “完全替代开发人员”；
- “百分之百发现安全漏洞”；
- “已经达到银行生产级要求”；
- “审计数据绝对不可篡改”；
- “任何代码都可以自动生成并上线”；
- “使用的 Agent 越多效果就一定越好”。

推荐使用更稳妥的表达：

- 辅助研发和审查人员；
- 降低重复劳动并提高问题发现概率；
- 已完成关键治理闭环的原型验证；
- 通过隔离、门禁和人工审批降低 AI 风险；
- 后续通过私有化部署和内部系统集成完成生产化加固。

## 十二、最终建议

项目下一阶段不应继续横向堆积普通功能，而应集中完成三件事：

1. **银行化**：补充权限分离、安全门禁、敏感数据保护和风险审批；
2. **证据化**：通过对照实验和量化指标证明效率与风险控制价值；
3. **场景化**：用转账、账户或权限接口的完整演示讲清楚业务价值。

最终让评委记住的核心信息应该是：

> AgentCollab 不是一个“更会写代码的 AI”，而是一套让 AI 能够在银行研发治理框架下安全工作的协作机制。
